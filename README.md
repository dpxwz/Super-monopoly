# Super Monopoly

一个本地网页大富翁游戏。没有后端，没有构建步骤。打开本地静态服务器后，2 到 4 名玩家可以同屏轮流操作。

## 当前规则

规则源文件是 `new_rules.md`。当前实现覆盖其中已经写清楚的规则；文档里标为“暂未制定”的内容不会被代码猜测补完。

已实现：

- 地图是 12×12 的外圈棋盘，总计 44 个格子。
- 第一格是“起始格”。玩家经过或停在起始格时获得 $200。
- 其余 43 个格子都是普通城市地块。
- 没有抵押、银行回购、铁路、机场、水电公司、税格、机会卡或其他特殊计价格。
- 每个城市拆成 10 股。每股代表 10%。
- 银行只持有尚未售出的初始股份。已售出的股份不会回到银行。
- 玩家落在银行仍持股的城市时，可以按 10% 为单位购买银行剩余股份。
- 玩家落在已有玩家持股的城市时，先按股份比例支付过路费，再处理是否购买银行剩余股份。
- 银行股份不收租。落地玩家不向自己持有的股份付租，但仍要向其他股东付租。
- 自由交易可在任意阶段发起。交易内容可包含现金、股份和合同。交易只有被明确接受后才生效，过期交易不能再接受。
- 合同包括免费通行权、继承权和投票支持。
- 免费通行权绑定具体股份。股份转手后，免费通行权仍跟随该股份。
- 继承权绑定具体股份。被绑定股份不能普通交易；当前持有人破产时直接转给继承权持有人。
- 投票支持合同会强制义务人在下一次匹配的新地产投票中按合同立场投票，投票结束后消耗次数。
- 玩家持有某地块至少 30% 股份时，是该地块主要持股人。
- 玩家在某色组所有地块都至少持有 30% 时，是该色组主要持股人。
- 建房要求目标地块没有银行股份。
- 完整拥有同色组，或作为同色组主要持股人且在目标地块至少持有 50%，可以直接建房或拆房。
- 同色组主要持股人在目标地块不足 50% 时，可以发起建房或拆房投票。至少 50% 股份投支持票时通过。
- 建房费用由目标地块所有玩家股东按股份比例承担；拆房返还费用同样按股份比例返还给所有玩家股东。
- 股东现金不足时，建设不会完成，游戏进入费用待支付状态。目标股份交易后，费用义务会跟随股份转移；当当前义务持有人都能付款时，建设完成。
- 自动破产已移除。租金等支付可以让当前玩家现金变为负数；该玩家不会自动结束游戏，而是卡在当前回合，直到现金恢复为正数，或主动选择破产。
- 主动破产会暂停正常流程，进入破产拍卖等待状态。股份不会自动回到银行。

当前保留为暂停或占位：

- 破产拍卖的参与者、起拍价、现金处理、失败处理还没有规则。
- pending trade 的复杂冲突处理还没有规则。当前只在接受交易时重新校验资产是否仍可用。
- UI 交易面板每边一次只选择一个地块的股份；规则层 API 支持把多份股份和多个合同放进同一笔交易。
- 同一股份重复绑定同类合同当前被阻止；免费通行权与继承权同时绑定、合同拍卖、合同继承仍未制定。
- 金额按 JS number 做比例计算，允许小数。`new_rules.md` 尚未规定四舍五入或最小货币单位。

## 运行

```bash
npm run serve
```

作用：在项目目录启动一个静态文件服务器，默认端口是 4173。

原因：`index.html` 使用 ES module 加载 `src/ui.js` 和 `src/game.js`。用 HTTP 服务打开可以避开浏览器对 `file://` 模块导入的限制。

避免的问题：直接双击 `index.html` 时，部分浏览器会因为 CORS 策略拒绝加载本地模块。

打开：

```text
http://127.0.0.1:4173
```

作用：访问本地游戏页面。

原因：页面、样式、逻辑都在仓库里，不需要外部 CDN。

避免的问题：离线或网络不稳定时，游戏仍然可以运行。

## 测试

```bash
npm test
```

作用：运行 `tests/game.test.js`，覆盖棋盘结构、股份初始化、按股购买、按股分租、自由交易、三类合同、主要持股人、建房/拆房投票、建房费用分摊、拆房返还分配、主动破产暂停、负现金回合锁定，以及无抵押/无特殊计价地块的回归规则。

原因：核心规则写在 `src/game.js`，DOM 渲染写在 `src/ui.js`。先测试规则，可以把复杂的股份、合同和投票计算固定下来，再让界面只负责调用规则层。

避免的问题：后续改按钮、文案或布局时，不容易把“已售股份不回银行”“股东仍向其他股东付租”“继承权阻止普通交易”等规则改坏。

```bash
npm run check
```

作用：用 Node 的语法检查确认 `src/game.js` 和 `src/ui.js` 能被解析。

原因：这个项目不依赖打包器，语法检查是最轻的静态检查。

避免的问题：少一个括号、错误的 import/export 这类问题会在打开浏览器前被发现。

## 文件结构

```text
index.html          页面骨架、购买股份表单、交易面板、合同面板、投票面板
src/game.js         纯游戏规则：股份、交易、合同、建房、投票、破产状态
src/ui.js           DOM 渲染与按钮事件，只调用 game.js 导出的规则 API
src/styles.css      12×12 外圈棋盘、股份条、面板、表单和响应式样式
tests/game.test.js  核心规则测试
new_rules.md        强化版规则原文
```

`src/game.js` 不直接访问 DOM。这样做的原因是规则可以在 Node 测试里运行。避免的问题是把界面状态和游戏状态混在一起后，改按钮或布局会误伤租金、交易、合同和建房计算。

`src/ui.js` 不复制规则。它只展示股份分布、合同、投票和待处理交易，并把用户操作交给 `game.js`。这样做的原因是同一条规则只写一次。避免的问题是界面按钮和测试里的规则判断不一致。
