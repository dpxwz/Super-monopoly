# Super Monopoly

一个本地网页大富翁游戏。没有后端，没有构建步骤。打开本地静态服务器后，2 到 4 名玩家可以同屏轮流操作。

## 当前规则

- 地图是 12×12 的外圈棋盘，总计 44 个格子。
- 44 个格子全部是普通城市地块，没有监狱、机会卡、税、车站、机场、水电公司或其他特殊格。
- 每个城市使用固定价格、固定租金表和固定建房费用。
- 城市分成 11 个色组，每个色组 4 个城市。
- 只有同时拥有同一色组的全部城市，才可以在该色组城市上建房。
- 踩到别人拥有的城市会按当前租金支付给拥有者。
- 玩家现金不足会破产，名下城市回到市场。
- 剩最后一名未破产玩家时游戏结束。

## 运行

```bash
npm run serve
```

作用：在项目目录启动一个静态文件服务器，默认端口是 4173。

原因：`index.html` 使用 ES module 加载 `src/ui.js` 和 `src/game.js`。用 HTTP 服务打开可以避开浏览器对 `file://` 模块导入的限制。

避免的问题：直接双击 `index.html` 时，部分浏览器会因为 CORS 策略拒绝加载本地模块。

打开：

```text
http://127.0.0.1:4173
```

作用：访问本地游戏页面。

原因：页面、样式、逻辑都在仓库里，不需要外部 CDN。

避免的问题：离线或网络不稳定时，游戏仍然可以运行。

## 测试

```bash
npm test
```

作用：运行 `tests/game.test.js`，覆盖 44 个普通城市地块、12×12 外圈地图、固定租金、买地、收租、经过地图原点奖励、破产，以及同色组建房限制。

原因：游戏规则写在 `src/game.js`，DOM 渲染写在 `src/ui.js`。先测规则可以让界面改动不影响核心计算。

避免的问题：后续调整棋盘文案或样式时，不容易把收租、买地、色组建房等规则改坏。

```bash
npm run check
```

作用：用 Node 的语法检查确认 `src/game.js` 和 `src/ui.js` 能被解析。

原因：这个项目不依赖打包器，语法检查是最轻的静态检查。

避免的问题：少一个括号、错误的 import/export 这类问题会在打开浏览器前被发现。

## 文件结构

```text
index.html          页面骨架和可访问标签
src/game.js         纯游戏规则，供测试和界面复用
src/ui.js           DOM 渲染与按钮事件
src/styles.css      12×12 外圈棋盘、面板、响应式样式
tests/game.test.js  核心规则测试
```

`src/game.js` 不直接访问 DOM。这样做的原因是规则可以在 Node 测试里运行。避免的问题是把界面状态和游戏状态混在一起后，改按钮或布局会误伤规则。

`src/ui.js` 只调用 `game.js` 导出的函数。这样做的原因是界面层负责显示和交互，规则层负责判断现金、城市、色组、回合。避免的问题是同一条规则在多个按钮里重复实现。
